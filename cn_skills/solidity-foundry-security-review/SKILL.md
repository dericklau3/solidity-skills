# 审查 Foundry 项目中的 Solidity 合约

## 核心工作流

### 回复前先理解请求

对于概念性问题，解释安全概念即可，不要假装已经审计过代码。对于审查请求，执行下面的流程。

### CRITICAL: 总是先阅读项目

在报告发现或建议修复前：

1. **搜索用户项目**中的合约、测试、脚本、接口、库、mock 和文档。
2. **先阅读相关文件**再下结论。当行为跨越文件边界时，也要阅读相邻合约。
3. **默认审查用户现有系统，而不是想象中的通用合约**。当用户说“审查安全”时，通常需要的是基于实际业务逻辑和调用路径的发现。

如果某个文件无法读取，明确暴露失败信息。报告尝试读取的路径，并说明为什么审查不完整。不要静默退回到通用 checklist。

### 基本规则：先重建业务逻辑，再找漏洞

多数严重 Solidity 问题是协议预期规则被破坏，而不是孤立的语法危险。

在给疑似问题定性之前，先建立：

1. **参与者** - 谁可以调用每条关键路径，以及这些权限依赖什么假设。
2. **资产和价值流** - 什么会移动、铸造、销毁、锁定、支付、记账或累计。
3. **状态机** - 存在哪些状态，如何转换，哪些转换应当不可能。
4. **会计模型** - 哪些总量、余额、份额、债务、奖励、限制或费率必须保持一致。
5. **信任假设** - 系统依赖哪些外部依赖、操作方、输入、时间假设或链下事实。

不要只依赖漏洞分类表。每个发现都应说明哪个业务规则、invariant 或信任假设被破坏。

### 库和依赖规则：验证项目中安装的源码

在标记或修复依赖 OpenZeppelin、Solady、Uniswap、Chainlink、代理库、Token 标准或其他依赖的行为前：

1. **定位项目中安装的依赖**：通过 `foundry.toml`、`remappings.txt`、`lib/`、`node_modules/` 或包配置确认路径。
2. **阅读项目实际使用版本的源码**，不要依赖记忆中的 API。hook、override 点、storage 和 initializer 要求都可能随版本变化。
3. **优先使用成熟库组件和文档化的扩展点**，而不是自定义修复逻辑；如果库已经提供 guard、role system、token extension、oracle check、proxy pattern 或 utility，应优先采用。
4. **不要把外部库源码复制进用户合约**作为修复。应通过 import、inherit、compose 或 configure 使用依赖。
5. 当出现 proxy、initializer、namespaced storage、gap、delegatecall 或升级脚本时，**把可升级性和 storage layout 作为一等审查范围**。

### 方法论

主要流程是**业务模型驱动的攻击路径分析**：

1. 检查项目文件并理解预期行为。
2. 检查已安装依赖源码，理解被 import 组件和集成假设。
3. 建立参与者、资产、状态和 invariant 的紧凑模型。
4. 通过 public entrypoint 追踪真实成功路径和失败路径。
5. 只有在映射到项目实际行为后，再检查通用 Solidity 风险。
6. 只报告基于文件、带攻击或失败路径、并有 Forge 测试思路的发现。

完整流程见 [业务模型和攻击路径分析](#业务模型和攻击路径分析)。

## 业务模型和攻击路径分析

这个流程用于审查 Foundry 项目，避免把任务降级成通用 checklist。

**前置条件：**始终先重建业务模型。

### Step 1: 确定范围和意图

1. 如果存在，阅读 `README.md`、`foundry.toml` 和项目文档。
2. 搜索 `src/`、`test/`、`script/` 以及相关接口或库。
3. 识别范围内合约，以及理解它们所需的相邻合约。
4. 阅读测试以推断预期行为，但不要假设测试完整或正确。
5. 当 import 或继承影响行为时，阅读 `remappings.txt`、`lib/`、`node_modules/` 和包配置。
6. 记录任何会阻碍完整审查的缺失上下文。

### Step 2: 建立协议模型

在寻找发现之前，写下你正在审查的模型：

- 外部参与者和特权参与者
- 用户行为和操作方行为
- 资产、余额、额度、债务、限制和其他会计变量
- 状态转换和终态
- 外部集成及其假设
- 时间、价格、随机数、签名或配置假设

用这个模型判断什么是“正确”。业务逻辑问题通常就是这个模型和实现之间的不一致。

### Step 3: 推导 Invariant 和失败条件

把模型转化为代码应保持的属性。

有用的属性类型示例：

- 用户不能获得超过其应得范围的价值。
- 总会计不能偏离实际资产或已记录义务。
- 状态转换不能跳过必要前置条件。
- 特权操作不能破坏用户保证，除非这是明确设计。
- 外部输入不能让过期、循环或不一致的值看起来有效。

这些只是示例，不是 checklist。应从项目自己的规则推导属性。

### Step 4: 通过 Public Entrypoint 追踪攻击路径

对于每个疑似问题，回答：

1. 哪个 invariant、业务规则或信任假设被破坏？
2. 哪个参与者可以用正常入口触达这条路径？
3. 哪个调用序列或状态变化会触发问题？
4. 影响是什么：资产、权限、会计漂移、拒绝服务，还是用户保证被破坏？
5. 最小安全修复是什么？
6. 什么 Forge 测试可以证明 bug 和修复？

如果无法描述可信的攻击或失败路径，不要把它作为强发现。降低置信度，或移到剩余风险。

### Step 5: 应用 Solidity 安全视角

业务模型清楚后，只应用与代码相关的视角：

- 访问控制和信任边界
- 重入和外部调用顺序
- 可升级性、delegatecall、初始化和存储布局
- 依赖集成：必需 override、hook、modifier、constructor、initializer、storage 和版本特定行为
- 签名验证、重放保护和授权
- 算术、取整、会计和状态转换正确性
- Token 和 NFT 集成行为
- 定价、新鲜度、滑点、随机数和时间假设
- 原生资产转账、退款、强制转入和低级调用
- storage、memory、calldata、delete 语义，以及链上秘密误解

不要把所有分类都倒进最终回答。它们用于找问题，最终只报告基于代码的结果。

### Step 6: 推荐最小修复和 Forge 测试

对于每个发现：

- 适用时优先推荐成熟库、项目已安装依赖或已验证模式，而不是自定义逻辑。
- 如果库组件是正确修复，指出 import、继承/组合变化、必需 override 或 initializer，以及任何 storage layout 影响。
- 不要把外部库源码复制进用户合约。
- 建议能恢复被破坏 invariant 的最小修复。
- 至少建议一个修复前失败、修复后通过的 Forge 回归测试。
- 当问题涉及会计、顺序或状态机时，补充边界测试或 invariant 测试建议。

## Foundry 审查指导

优先基于项目文件，而不是抽象推理。常用验证命令包括：

- `forge test -vvvv`
- `forge test --match-contract <ContractName> -vvvv`
- `forge test --match-test <TestName> -vvvv`

当项目依赖 fork 状态或外部系统时，指出任何会限制审查置信度的缺失 RPC、fork block、环境变量或依赖项。

## 输出约定

默认审查输出应遵循以下结构：

### 1. 范围和模型

- 已审查文件
- 相关依赖和相邻合约
- 参与者、资产、主要流程和关键假设
- 审查中使用的 invariant 或业务规则

### 2. 发现

按严重程度排序：`Critical`、`High`、`Medium`、`Low`。

每个发现包括：

- 严重程度
- 受影响代码
- 被破坏的 invariant 或业务规则
- 攻击路径或失败路径
- 影响
- 修复方案
- 建议的 Forge 测试
- 需要时给出置信度

### 3. 未发现实质性问题的情况

如果没有发现实质性问题，明确说明。

然后包括：

- 剩余风险
- 未覆盖的攻击面
- 仍需测试的业务规则或 invariant

## 回复风格

- 简洁直接
- 先给发现，不要先铺背景
- 不要把回复变成通用安全百科
- 不要要求用户检查你自己可以检查的文件
- 清楚说明不确定性和假设
- 如果代码看起来可以接受，但业务模型或测试面薄弱，要明确指出
