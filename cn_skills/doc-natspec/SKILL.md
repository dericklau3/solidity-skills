# 为 Solidity 合约编写和修复 NatSpec

## 核心工作流

### 编辑前先理解用户请求

当 Solidity 开发已经完成，且用户希望修复、补全或检查 NatSpec 覆盖情况时，使用这个 skill。

默认情况下，检查 Foundry 项目 `src/` 目录下的合约，除非用户指定了明确的文件。

## 先阅读项目

在编辑 NatSpec 之前：

* 如果存在 `foundry.toml`，阅读它
* 在 `src/` 中搜索相关 Solidity 合约
* 默认排除 interfaces 和 libraries，除非用户明确要求为它们编写文档
* 在需要理解语义时，阅读相邻测试和辅助合约

如果某个必需文件无法读取，需要明确说明，不要假装文档工作已经完整完成。

## 默认范围

默认检查：

* `src/` 下的 Solidity 合约
* 用户指定的 Solidity 文件

默认不检查：

* interfaces
* libraries
* 纯测试文件
* 无关的历史文件

## 必需覆盖范围

以下内容必须有 NatSpec：

* 范围内的每个 `public` 函数
* 范围内的每个 `external` 函数
* 范围内每个复杂的 `internal` 函数
* 范围内每个复杂的 `private` 函数
* 范围内每个对外有意义的 `struct`
* 范围内每个对外有意义的 `event`

非常小的 helper、明显的透传 wrapper，以及简单的内部 plumbing，不需要强制添加 NatSpec，除非用户明确要求对 internal 内容做完整覆盖。

## 质量标准

### 对于函数

要求：

* 必须有 `@notice`
* 当行为、限制、权限、副作用、不变量或边界情况不是显而易见时，必须有 `@dev`
* 每个输入参数都必须有一个对应的 `@param`
* 每个返回值都必须有一个对应的 `@return`
* 确保注释文本与实现一致
* 拒绝只重复标识符、信息量低的填充式注释
* 避免记录实现并不保证的行为

### 对于 struct 声明

要求：

* 解释该 struct 表示什么
* 分别解释重要字段
* 重点描述字段含义，而不是重复字段类型

### 对于 event 声明

要求：

* 解释该 event 在什么时候、为什么会被触发
* 分别解释重要参数
* 让链下观察者和集成方能够理解该 event

注释必须准确、具体，并且对文档读者和审查者都有实际帮助。

通过 `forge doc` 只是最低验证标准，不是质量标准。

## 编辑规则

### 直接修复明确缺口

如果可以从代码、命名、测试和调用路径中清楚理解语义，则直接添加或改进 NatSpec 注释。

### 遇到歧义时停止

如果无法可靠确定某个函数、字段或 event 参数的含义，不要编造文档。

需要指出具体对象，解释歧义点，并保持该部分不变。

### 保持范围收敛

不要把 NatSpec 工作变成重构、重命名清理，或整个仓库的风格重写。

## 验证

编辑后运行：

```bash
forge doc
```

如果失败，需要说明失败原因看起来属于以下哪一种：

* 新增或修改的 NatSpec 导致的问题
* 已存在的编译或配置问题
* 与当前文档修改无关的更大范围项目问题

不要在没有报告 `forge doc` 结果的情况下声称工作已经完成。

## 输出要求

默认最终输出应包括：

* 已审查的文件
* `forge doc` 结果
* 剩余的文档风险或语义不清的地方

## 回复风格

* 直接
* 基于具体文件
* 不编造含义
* 不要把“最低标签覆盖率”误认为“高质量文档”
