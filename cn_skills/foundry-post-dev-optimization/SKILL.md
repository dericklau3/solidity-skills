# Solidity 合约开发后的优化

## 核心工作流

### 编辑前先理解请求

当 Solidity / Foundry 开发已经完成，并且用户希望对现有合约进行一次聚焦的优化检查时，使用这个 skill。

这个 skill 适用于开发后的 Gas、代码结构和可维护性优化，不适用于初始功能实现。

## 先阅读项目

在进行优化编辑之前：

* 如果存在 `foundry.toml`，阅读它
* 搜索 `src/` 目录下范围内的 Solidity 合约
* 默认排除 interfaces，除非用户明确要求包含它们
* 在需要理解预期行为时，阅读相邻的测试、mock、fixture、script 和辅助合约
* 检查仓库是否已经存在 benchmark、snapshot、gas-report 或其他优化验证流程

如果无法读取必要文件，需要明确说明，不要假装优化检查已经完成。

## 默认范围

默认检查：

* `src/` 目录下的 Solidity 合约

默认排除：

* interfaces

如果用户指定了特定合约或文件，则缩小到对应范围。

## 优化类别

检查以下方面的优化机会：

* Gas 效率
* 代码结构
* 可维护性

不要机械式优化。相比表面改动，优先选择有实际意义且行为安全的改进。

## 优化等级

### 1. 安全的直接编辑

当语义清晰时，可以直接 patch 低风险优化。

常见示例：

* 在安全的情况下，将 `memory` 改为 `calldata`
* 将 revert string 改为 custom error
* 缓存重复读取的 storage 值或长度
* 删除冗余计算、变量或分支
* 抽取小型 helper，以提升清晰度，同时不改变行为

### 2. 有条件的编辑

只有在行为清晰且验证强度足够时才 patch。

常见示例：

* `unchecked`
* storage packing
* loop 优化
* state variable 重新排序
* 减少状态转换和外部调用周围的重复读取

只有在满足以下条件时才 patch：

* 可以从代码和上下文清楚理解语义
* 不会违反 storage layout 或 upgradeability 假设
* 不会破坏外部集成预期
* 现有验证足够强，可以支撑这个改动

### 3. 只作为建议的编辑

除非用户明确想要更激进的优化，并且该改动有充分理由，否则以下内容只作为建议保留。

常见示例：

* assembly
* 改变 ABI 形态的结构性重写
* 低价值 Gas 优化，但会明显降低可读性
* 只为了理论收益而进行的大范围多合约重构

对 assembly 要比其他优化技术更加保守。

## 编辑规则

### 先 review，再 patch

在编辑前先识别优化机会，确保改动集是有意图且范围受控的。

### 直接 patch 明确收益

如果可以从实现、测试、命名和仓库上下文中清楚理解预期行为，则可以直接应用安全或理由充分的优化。

### 遇到歧义就停止

如果某个优化可能改变语义、storage 预期、集成假设，或者以非平凡方式影响可读性，不要猜测。

应该指出该优化机会，解释风险，并把它保留为建议。

### 保持范围收敛

不要把优化检查变成大型重构、安全审计、功能重写或代码风格清理。

## 验证

优先使用项目已有的最强验证路径。

优先级：

1. 现有 benchmark script 或 CI 性能基线
2. 现有 gas snapshot 或 gas 对比流程
3. 现有 gas-report 流程
4. fallback 到 `forge test`

如果可用，优先进行优化前后的对比，而不是只做单边测量。

如果验证失败，需要说明失败看起来是由以下哪类原因导致：

* 优化改动本身
* 仓库中已有的问题
* 与优化 patch 无关的项目配置或依赖问题

如果没有报告验证结果，不要声称优化工作已经完成。

## 输出约定

默认最终输出应包含：

* 已 review 的合约和相关文件
* 优化范围
* 已实现的优化
* 建议但跳过的优化
* 风险说明
* 验证结果

优化总结需要分为：

* Gas changes
* Code-structure changes
* Maintainability changes

对于跳过的项目，需要包含原因：

* 语义不清晰
* upgradeability 或 storage layout 风险
* 需要激进的 assembly
* 收益较低，不值得牺牲可读性
* 验证信心不足

## 响应风格

* 直接
* 基于具体文件
* 优先追求真实价值，而不是机械式微优化
* 不要编造 Gas 改进
* 不要模糊优化、测试和安全审计之间的边界
